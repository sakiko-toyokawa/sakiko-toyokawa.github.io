Codex-RS 学习报告：run_turn() 进入 loop 前的准备阶段

Codex-RS 学习报告：run_turn() 进入 loop 前的准备阶段

一、为什么要单独研究”准备阶段”

在 codex-rs 里，run_turn() 是最核心的一条执行链。它负责把”一次用户输入”真正变成”一轮模型交互”，并在需要时继续驱动工具调用、收集工具结果、再把这些结果送回模型，直到本轮对话结束。从外部看，这像是一次完整问答；但从内部看，它更像一个小型状态机：先确定上下文，再准备运行条件，再进入采样循环，最后再收束结果。

很多人第一次读 run_turn() 时，容易把注意力放在后面的 loop 上，因为那里有模型流式输出、工具调用、重试、错误恢复这些最显眼的逻辑。但如果只盯着 loop，会产生一个错觉：好像用户一发消息，系统就立刻开始向模型采样。实际上并不是。真正的 run_turn() 在进入 loop 之前，已经做了大量”让模型这次能正确工作”的准备，这些准备既决定了模型能看到什么，也决定了它不能看到什么；既影响 token 成本，也影响权限边界；既关系到性能，也关系到安全。

因此，本文不讨论整个 run_turn() 的所有细节，而是聚焦一个更窄、但也更关键的问题：在正式进入采样 loop 之前，Codex-RS 到底做了哪些准备工作，它们为什么必须在前面完成？

如果把 run_turn() 比作一次正式发车，那么 loop 相当于列车开始运行，而本文要分析的是列车发车前的站台调度：线路有没有锁定、乘客有没有上车、行李有没有安检、车厢内的广播是否准备好、司机能看到的仪表盘是否完整。这些步骤看上去不如”开车”本身热闹，但它们才决定了这趟车能不能稳定、安全、低成本地跑起来。

从源码位置看，本文讨论的主战场主要在：

  • core/src/session/turn.rs
  • core/src/session/turn_context.rs
  • core/src/session/mod.rs
  • core-skills/src/injection.rs
  • core/src/plugins/injection.rs

这些文件共同完成了 turn 级配置冻结、上下文压缩、上下文注入、skill/plugin 解析与注入、hook 执行、用户输入落库，以及进入 loop 前的运行态初始化。

为了把这段代码讲清楚，本文采用”按真实调用链展开”的方式，而不是抽象成一套脱离源码的架构图。因为 codex-rs 的准备阶段不是几个并列模块简单拼接，而是一条前后有严格顺序的流水线：前一步的输出，往往正是后一步的输入。

二、准备阶段的起点：TurnContext 其实在 run_turn() 之前就已经生成

研究 run_turn() 之前，必须先澄清一个常见误解：本轮 turn 的配置快照，并不是在 run_turn() 里面现算出来的，而是在 run_turn() 被调用之前，就已经由 TurnContext 固定下来了。

这件事非常重要，因为后面整个准备阶段都依赖这个前提：run_turn() 不是临时从 Session 的全局状态里”边读边取”配置，而是接收一个已经构建完毕、在本轮生命周期内应当视为稳定不变的 TurnContext。

调用链大致是这样的：

  1. 外层提交用户输入。
  2. Session 根据当前配置和这次更新，生成新的 turn 配置。
  3. 通过 new_turn_from_configuration() 构造 TurnContext。
  4. RegularTask::run() 再把这个 TurnContext 传给 run_turn()。

在 core/src/session/turn_context.rs 中，new_turn_from_configuration() 做了几件关键的事情。

第一，它确定本轮真正生效的 cwd 和环境。这里不是简单取 session 级配置，而是先看本轮是否有显式 turn environment；如果有，就优先使用 turn environment 的工作目录，否则再退回 session 的 cwd。这说明 Codex-RS 从设计上就允许”同一个 session 内，不同 turn 拥有不同执行环境”。

第二，它基于 SessionConfiguration 派生出 per_turn_config。这一层的意义是把”全局可变配置”变成”本轮已解析配置”，后续模型信息、插件、skill、工具能力、权限策略都以这个 per-turn 视角为准。

第三，它会立刻把当前 turn 的审批策略和权限配置同步给 mcp_connection_manager。这意味着外部 MCP 相关能力也必须和本轮权限边界保持一致，而不是沿用旧状态。

第四，它根据当前模型名和 per-turn config 去 models_manager 里查 model_info。这一步的结果非常关键，因为后面是否支持并行工具调用、上下文窗口大小、是否启用 auto compact、输入模态等，都来自 model_info。

第五，它根据配置加载本轮可见的插件，再从插件导出有效的 skill roots，然后交给 skills_manager 计算本轮可用 skills。这里已经能看出一个很典型的 Codex-RS 设计特点：skills 并不是一个独立于插件系统的孤立层，它和插件提供的能力空间是联动的。

最后，make_turn_context(...) 会把这些信息收束成一个完整的 TurnContext：其中包含模型信息、配置、权限、环境、plugin/skill 结果、telemetry、输出 schema、session source 等等。到这一步，run_turn() 才拿到本轮真正的执行上下文。

这就是为什么说 TurnContext 本质上是一份”turn 级快照”。它的核心价值不是为了封装数据，而是为了隔离变化。Session 在整个生命周期内是活的，配置可能继续被改；但某一轮 turn 一旦开始执行，就不能在半路换模型、换权限、换工作目录，否则模型看到的世界会前后不一致，工具层的安全边界也会失真。

这和 thread_config_snapshot() 形成了一个对比。thread_config_snapshot() 位于 core/src/session/session.rs，更像是一个对外展示或查询用的简化快照，它返回 model、approval policy、permission profile、cwd、personality 等关键信息，适合状态栏、线程视图或其他只读场景。而 TurnContext 则是运行态真正使用的完整版上下文，它不仅更全，而且生命周期也更严格：一旦创建，就代表这轮 turn 的执行基线已经锁定。

因此，从逻辑上说，run_turn() 的准备阶段其实不是从零开始，而是从一个已经冻结好的 TurnContext 开始。这是后面所有步骤能稳定运行的前提。

三、第一道闸门：pre-sampling compact 先判断上下文是否已经太大

run_turn() 开始后，最早执行的重要步骤之一，是 pre-sampling compact：

这一段出现在一切正式注入和采样之前。也就是说，系统还没把本轮新的上下文更新和新的用户输入正式记录进模型历史时，就先检查：现有历史会不会已经太大，大到需要压缩？

这一步的判断依赖模型的 auto_compact_token_limit()。代码中先从 turn_context.model_info 取出 auto compact 阈值；如果模型没配置这个阈值，就退回 i64::MAX，相当于”默认不触发自动压缩”。这说明 compact 不是一个全局一刀切的策略，而是与模型能力绑定的。

从机制上说，compact 的目标不是”删除历史”，而是”重写历史”。它会收集历史消息，尤其是用户与助手之间那些占 token 很多、但不需要原样逐字保留的部分，再通过 compact 逻辑把长历史总结成更短的表示。这样做的核心收益有两个：

  1. 让上下文重新回到模型可接受的范围内。
  2. 尽量保留对后续推理仍然有价值的语义信息。

这里的设计重点，不是摘要这件事本身，而是压缩发生的时机。它被放在准备阶段最前面，是因为后续还要继续往历史里塞东西：context injection、skill 注入、plugin 注入、用户新输入、额外上下文，这些都会进一步增加 token 负担。如果不先做 compact，后面再构建 prompt 时就更容易触发 context window 问题。

不过，源码里有一条非常值得注意的 TODO，直接写在 run_turn() 附近。意思是：当前 pre-turn compaction 发生得比较早，它是在新的 context update 和新用户消息记录之前运行的，因此现在的判断还没有把”即将新增的这些内容”算进去。作者已经意识到，未来更理想的实现应该是预估”本轮待注入内容”会增加多少 token，再更提前、更准确地判断是否要 compact。

这条 TODO 很有价值，因为它揭示了一个现实：Codex-RS 当前的 compact 机制已经实用，但并不是数学上最精确的。它优先解决了”在大多数情况下先别爆上下文”的问题，而不是一步做到最优。

压缩之后还有一个经常被忽视、但很关键的联动动作：如果本轮之前已经预热过 prewarmed_client_session，而 compact 又真的发生了，那么系统会调用：

原因很直接。预热 session 的价值在于提前建立或复用与模型端的通信状态，例如 websocket 会话、上一次响应关联信息、粘性路由状态等。但 compact 一旦改写了历史，原来那个预热 session 所假设的”上下文基线”就不一定还成立。如果继续沿用旧会话，模型端和本地记录可能已经对不上。因此，compact 发生后必须重置 websocket session，确保后续真正采样时不会带着过期状态继续跑。

从工程视角看，这一步说明 Codex-RS 对”会话复用”是谨慎的：可以为了性能尽量复用，但一旦前提条件变化，就必须果断丢弃，优先保证一致性。

所以，pre-sampling compact 的真正意义，不只是”省 token”。它还是整个准备阶段的第一道闸门：先确保现有历史处于一个可继续增量构建的状态，后面所有准备动作才有地方落脚。

四、上下文注入：模型不是天然知道当前环境，系统必须显式告诉它

完成 compact 检查之后，run_turn() 进入下一步：

这行代码的名字很长，但也恰好暴露了它的两个职责：

  1. record_context_updates：把本轮模型需要知道的上下文变更，记录进对话历史。
  2. set_reference_context_item：把当前 turn 的上下文快照设为后续 diff 的参考基线。

为什么一定要有这一步？因为模型虽然会收到 prompt，但它并不会天然知道当前 session 的工作目录、权限边界、审批策略、协作模式、额外开发者指令、用户指令、人格设定等运行条件。如果不把这些内容显式注入到模型可见历史中，模型就是在一个”只看得到用户消息、看不到操作环境”的半盲状态里工作。

4.1 首轮全量注入，后续按 diff 注入

record_context_updates_and_set_reference_context_item() 的核心逻辑在 core/src/session/mod.rs。它首先从 session state 中取 reference_context_item()。这个字段可以理解成：”模型上一次已经被明确告知的 turn 配置基线是什么”。

然后逻辑分成两路：

  • 如果 reference baseline 不存在，说明这是第一次注入，或者中途被重建过基线，那么走 build_initial_context(turn_context)。
  • 如果 baseline 已经存在，说明模型之前已经见过一个版本的上下文，那么只需要走 build_settings_update_items(...)，生成本轮相对上轮的增量变化。

这是一种很典型的”首轮全量、后续增量”的上下文管理策略。它背后的收益非常直接：减少重复 token。

首轮时，模型需要一次性知道完整背景，因此必须全量注入。这个全量上下文通常包括：

  • base instructions
  • developer instructions
  • user instructions
  • 当前工作目录与环境
  • 权限与审批配置
  • collaboration mode
  • personality 相关说明
  • 其他会影响模型行为的系统级运行条件

但如果第二轮、第三轮、第四轮这些配置都没变，那么每轮都原封不动再发一遍，就会造成巨大的 token 浪费。Codex-RS 通过 reference baseline 机制，把这种重复开销降到了最低：只有配置变了，才把变化重新告诉模型。

4.2 这一步不仅写历史，也持久化运行基线

很多人在阅读这段代码时，会把注意力只放在 record_conversation_items() 上，以为它只是”把上下文消息塞进会话历史”。其实还有另一件同样重要的事：

也就是说，系统不仅把模型可见的上下文差异记进 history，还会把当前 turn 的 TurnContextItem 单独持久化到 rollout 里。源码注释写得很清楚：即使这一轮没有新的 model-visible diff，也仍然要持久化一个 TurnContextItem，这样在 resume、lazy replay、重建 durable baseline 时，系统仍然可以恢复出”这轮真实生效的上下文状态”。

这说明 record_context_updates... 不只是 prompt 层的注入函数，它同时还是恢复能力的一部分。它把”模型看到的变化”和”系统内部需要记住的状态基线”同时维护起来。

4.3 为什么 diff 基线一定要前移

函数最后还有一步：

即使这一轮没有任何新的上下文项被注入，它也会更新内存中的 diff baseline。为什么？因为”是否需要给模型补发上下文”与”系统当前真实配置状态是什么”是两个相关但不完全相同的问题。某些场景下，这一轮没有产生新的 model-visible 文本，但 turn 的运行态仍然已经切换到了新的配置。如果不更新 baseline，后续 diff 就会拿旧状态做比较，导致重复注入或者漏注入。

换句话说，这里的 baseline 不是一个”模型历史里上次发了什么文本”的简单缓存，而是一个”下一轮应该以什么状态为参照”的运行基准。

从整体上看，这一阶段完成的是一项基础设施工作：把 session 内部的真实配置状态，转化成模型能够理解、并且尽量低成本维护的一套显式上下文。这是后续任何高质量采样的前提。

五、skill 与 plugin 注入：把用户显式点名的能力暴露给模型

上下文基线建立好之后，run_turn() 开始处理本轮显式提及的扩展能力。这里主要分成 plugin 和 skill 两条线，但它们不是完全独立的，而是共享一部分前置数据，例如可用插件、MCP 工具、可访问 connectors。

5.1 先确定本轮有哪些插件和 app 能力真正可见

代码首先根据当前 turn 的配置，向 plugins_manager 请求当前可用插件：

接下来会解析用户输入中显式提到的插件。这里处理的是两类显式引用：结构化的 plugin://... 路径，以及文本中的插件提及。系统只会对当前 session 已启用的插件做解析，而不会凭空让一个未启用插件进入本轮 prompt。

随后，如果本轮开启了 apps，或者虽然没开 apps 但用户显式提到了插件，系统就会去 mcp_connection_manager 里列出当前所有 MCP tools。这里的设计很细：即使 app tools 平时默认隐藏，只要用户明确点名了一个插件，系统也愿意去拿它当前可用的 MCP/app inventory，因为接下来它要把”这个插件现在到底能干什么”告诉模型。

然后代码还会构造 available_connectors。这一步不是简单读取插件声明，而是经过几层筛选：

  1. 插件声明了哪些 app。
  2. 当前 MCP 环境实际上暴露了哪些可访问 app。
  3. 两者合并后，再根据 turn config 里的启用状态做过滤。

最后留下的，才是本轮真正对模型有意义的 app/connector 集合。这个过程体现出 Codex-RS 的一个基本原则：模型能被告知的能力，应当以当前会话真实可达能力为准，而不是只看静态声明。

5.2 skill 提及解析：不是所有名字都能直接命中

skill 注入逻辑由 collect_explicit_skill_mentions() 和 build_skill_injections() 组成。

collect_explicit_skill_mentions() 位于 core-skills/src/injection.rs。它不是一个简单的字符串查找，而是一套有选择规则的解析流程：

  • 先处理结构化的 UserInput::Skill，按路径做精确匹配。
  • 对文本输入，再扫描 $skill-name 这类 mention。
  • 解析时会跳过 disabled skills。
  • 会同时做去重。
  • 对 plain name 的匹配还要考虑歧义；如果多个 skill 名字冲突，或者与 connector slug 冲突，就不会轻率匹配。

这部分代码非常说明问题。Codex-RS 并不想把”用户写了一个看起来像 skill 名称的词”直接当成高可信指令，而是尽量要求显式且可消歧的引用。这能降低误注入 skill 的风险。

5.3 skill 文件真正注入的，是 SKILL.md 正文

解析出 mentioned_skills 后，build_skill_injections() 才真正去读 skill 文件。它会为每个显式提及的 skill 找到对应文件系统，再读取 path_to_skills_md 指向的 SKILL.md 文本，把结果封装成：

  • name
  • path
  • contents

如果读取失败，不会让整轮 turn 崩掉，而是把错误转成 warning。这个 warning 后面会通过 session 事件通道广播出去，让前端或其他订阅方决定如何呈现。

成功读取后的 skill 注入不会立刻”变成 Prompt”，而是先被包装成 ResponseItem。值得注意的是，这里用的是 SkillInstructions，它实现的 ContextualUserFragment 角色是 user，并带有 <skill>...</skill> 标记。这意味着 skill 更像是”用户在当前回合显式附带的一份额外操作说明”，而不是系统级 developer 指令。

这种角色设计很微妙，也很合理。skill 本身通常表达的是一种任务专长或专门工作流，往往是用户显式要求系统临时引入的能力，因此放在 user 语义层更贴近真实意图。

5.4 plugin 注入：不灌整份插件文件，而是灌能力摘要

plugin 的处理方式和 skill 不同。build_plugin_injections() 位于 core/src/plugins/injection.rs，它不会去读取某个”插件全文说明文档”塞给模型，而是根据当前显式提及的插件、当前可见 MCP server、当前可用 app，生成一段developer hint。

这些 hint 大致会告诉模型：

  • 这个插件有哪些可见 MCP servers。
  • 这个插件是否带有 skill 前缀。
  • 这个插件对应哪些当前启用的 apps。

最后这段内容被包装成 PluginInstructions，角色是 developer。这意味着 plugin 注入表达的是”系统层面对当前能力面的说明”，而不是用户要求模型必须执行的一段内容。

这和 skill 的区别很关键：

  • skill 更像用户显式附带的专业化提示，放在 user 侧。
  • plugin 更像系统告诉模型”你现在可以调哪些插件能力”，放在 developer 侧。

5.5 注入顺序与历史落点

在 run_turn() 里，skill 和 plugin 的结果并不是马上组装成最终 prompt，而是先转成 ResponseItem，然后通过 sess.record_conversation_items(...) 写进 conversation history。

顺序上，代码是在记录用户输入和 additional contexts 之后，再把 skill items 和 plugin items 记进历史。这一点值得在理解上保持准确：这些注入内容的直接落点是 session history，而不是某个一次性内存 prompt 缓冲区。

这也是为什么后面 loop 中每次重新构建 prompt 时，可以直接从最新历史拿到这些内容。准备阶段负责”把该存在的上下文写进历史”；采样阶段负责”按需把当前历史读出来组装 prompt”。两者职责清晰分离。

六、hook、用户消息与附加上下文：真正把”这轮输入”登记进系统

如果说前面的 compact、context injection、skill/plugin 解析主要是在为模型准备环境，那么接下来这部分逻辑就是在为”本轮输入本身”完成入场手续。

6.1 session start hook 先跑一遍

在记录用户消息之前，run_turn() 会先调用：

这意味着某些 session start hook 可能会延迟到真正有 turn 运行时才补执行。如果这些 hook 决定中止流程，那么本轮 turn 会直接结束，不再往下走。

这里体现出一个设计取向：hook 不是 loop 中的附属玩具，而是正式执行链的一部分。只要 hook 可能影响本轮是否继续，它就必须在采样前跑完。

6.2 user prompt submit hook 在消息入历史前拦截

接着，如果本轮 input 非空，系统会运行 run_user_prompt_submit_hooks(...)。它拿到的是本次用户消息构成的完整 message 文本，而不是拆散后的 token 片段。

这一层 hook 可以返回 should_stop 和 additional_contexts。如果 should_stop 为真，说明 hook 已经决定阻止本轮继续执行，那么系统只记录额外上下文后就返回，不再进入后续采样。

这一步的顺序也非常重要：hook 先看用户输入，再决定是否让这条输入真正进入主历史。 这让 submit hook 可以实现诸如预检查、策略拦截、输入增强等功能，而不会因为消息已经正式入历史而难以回头。

6.3 用户输入正式写入 conversation history

如果 submit hook 没有阻止本轮，系统才会调用：

这一步既记录用户 prompt，也触发 turn item 级别的事件。换句话说，到这里为止，”本轮用户确实说了什么”才真正成为 session 历史的一部分。

这也解释了为什么前面 compact TODO 会提到”当前 compact 发生在新的用户消息记录之前”。因为从源码顺序看，用户输入确实还没在最早阶段正式入历史。

6.4 additional contexts 和 connector selection 的合并

submit hook 返回的 additional_contexts 不会丢掉，而是通过 record_additional_contexts(...) 再写入系统。与此同时，代码还会把显式提及的 app/connectors 与从 skill items 中识别出来的 connector 使用意图合并，然后更新 session 级 connector selection。

这一部分看似零碎，实际上非常重要。因为它说明：用户这轮输入，不只是”一段文本”，还可能隐含”我想用哪个 app””我明确打开了哪些 connector””hook 给这轮补充了哪些额外上下文”。这些附带信号如果不在采样前统一落账，模型后面看到的 prompt 就只是一条裸用户消息，丢掉了很多可执行上下文。

6.5 previous turn settings 的更新

如果本轮确实有新输入，系统还会更新 PreviousTurnSettings，保存本轮模型与 realtime_active 状态。源码注释写得很清楚：这个 baseline 只在 regular user-turn 路径更新，这样 standalone 的 compact、shell、review 之类任务不会错误地污染后续”真实用户轮次”的设置基线。

这说明 Codex-RS 对”什么才算真实用户 turn”有严格区分，而不是把所有任务都混在一起做差分。

整体来看，这一大段逻辑的作用可以概括为一句话：把这轮输入从一个外部请求，转化成系统内部已经登记、已经附带上下文、已经可被模型消费的正式 turn 历史。

七、Prompt 什么时候真正组装：不是现在，而是在后面的采样 loop 中动态构建

很多文章讲到这里，会顺手把”技能/插件注入 + Prompt 组装”并列成同一个步骤，这样说并不完全错，但如果要严格对源码负责，就必须更精确一点：准备阶段确实完成了 prompt 需要的材料准备，但真正的 Prompt 结构体不是在这一步一次性组好的，而是在后面的 run_sampling_request() loop 中动态构建。

build_prompt(...) 定义在 core/src/session/turn.rs 靠后的位置：

这个函数本身很直白：它把当前历史、工具 schema、并行工具开关、base instructions、personality、输出 schema 等封装成一个 Prompt。但关键不在函数内容，而在它的调用时机。

在 run_sampling_request() 内部，系统会先构建 ToolRouter，再进入一个 loop。每轮 loop 都会做：

  • 如果这是第一次采样，就用传入的 input。
  • 如果不是第一次，就重新从 sess.clone_history().await.for_prompt(...) 拉最新历史。
  • 再调用 build_prompt(...) 生成当轮 prompt。

这意味着 prompt 是按回合即时构建的，而不是在准备阶段做一次、后面一直复用。这样设计的好处非常明显：

  1. 一轮 turn 内如果发生了工具调用，历史会变，新一轮 prompt 需要自动带上这些新结果。
  2. skill/plugin/context/user message 都已经在准备阶段写入 history，后续 prompt 不需要再维护一份平行缓存，只要读当前历史即可。
  3. prompt 构建逻辑与历史维护逻辑解耦，系统状态更容易保持一致。

因此，更准确的说法应该是：

  • 准备阶段负责把 prompt 所需的上下文材料写入系统历史，并完成运行态初始化。
  • 采样阶段在 loop 中按需从最新历史读取这些材料，动态构建真正送往模型的 Prompt。

这个区分非常值得强调，因为它直接体现了 Codex-RS 的架构风格：不是”先攒一个巨型 prompt 缓冲区再用”，而是”维护一份可重放、可增量更新的历史，再按需投影成 prompt”。

八、进入 loop 前的最后初始化：把执行态对象准备好

完成前面的上下文、消息、能力与 hook 处理之后，run_turn() 才进入最后一组”发车前检查”：

这几行常常被一眼带过，但它们实际上标志着准备阶段即将结束、执行阶段即将开始。

last_agent_message 用来记录本轮助手最终对外返回的最后一段消息；stop_hook_active 则参与后续停止逻辑的控制。它们属于 loop 运行中的状态变量。

TurnDiffTracker 则更有代表性。它负责追踪本轮执行过程中产生的文件变更 diff。虽然它在代码层面可能跟随 task 生命周期，但从用户语义上看，它服务的就是”这一轮交互造成了哪些变更”。因此它必须在 loop 启动前就创建好，这样后续工具执行、文件修改、输出总结都能共享同一份 diff 视图。

client_session 则是模型通信的真正入口。如果前面已经有预热好的 client session，就复用；否则新建一个。注意这个对象是 turn 级 的，不是整个 session 级永久单例。原因也很合理：同一轮 turn 内的多次采样、重试、工具往返，适合共享同一条模型通信会话；但跨 turn 之后，配置、历史、上下文都可能变化，没必要盲目强绑成全局长生命周期对象。

最后一个变量 can_drain_pending_input 也很有意思。它的初值取决于这轮 input.is_empty()。如果这轮已经有新的用户输入，系统不会一上来就先排空 pending input，而是优先让模型看到最新鲜的用户消息；只有在没有新输入时，才允许直接处理积压的 pending input。这个细节说明 Codex-RS 在 turn 调度上很注意”新鲜输入优先”的交互体验。

到这里，run_turn() 才真正具备了进入采样 loop 的所有条件：

  • 本轮配置已经冻结成 TurnContext
  • 历史大小已经过 compact 检查
  • 模型可见上下文已经注入
  • 显式 skill/plugin 能力已经转成历史项
  • 用户输入和额外上下文已经落账
  • hooks 已经执行完毕
  • 模型通信 session 与 turn diff tracker 已就绪

也就是说，loop 还没开始，但一轮 turn 在工程意义上其实已经”准备完毕”了。

九、如何理解这整个准备阶段的设计价值

把这段代码连起来看，会发现 Codex-RS 的准备阶段并不是若干随机散落的前置步骤，而是围绕一个核心目标展开：在真正采样之前，把本轮 turn 压缩成一个一致、稳定、低成本、可恢复、可执行的运行面。

这个目标至少体现在五个层面。

第一，是配置一致性。TurnContext 先冻结，避免执行过程中 session 配置漂移导致模型前后看到不同世界。

第二，是上下文可控性。通过 pre-sampling compact 和 context diff 注入，系统既控制了 token 成本，也控制了模型真正看到哪些运行条件。

第三，是能力显式化。skill 和 plugin 不会隐式神秘出现，而是要求用户显式提及、系统显式解析、最后显式写入历史。这让模型的能力面更加可追踪。

第四，是恢复与追溯能力。TurnContextItem 被持久化到 rollout，不只是为了当前这一轮能跑，还为了未来 resume、replay、分析时能重建当时的运行基线。

第五，是交互秩序。hooks、用户消息、additional contexts、pending input、connector selection 都在进入 loop 前排好顺序，避免采样时一边跑一边补上下文造成状态混乱。

如果没有这些准备动作，理论上系统也许还能”勉强发出一次模型请求”，但结果会很差：上下文容易超限，模型对当前环境认知不足，权限边界不清，用户显式提到的能力无法可靠注入，断线重试和 replay 也难以保持一致。换句话说，真正让 run_turn() 成为工程化对话循环的，不是后面的 loop 本身，而是前面这一整套准备机制。

十、结语：准备阶段决定了 loop 能否”正确地开始”

很多系统把模型调用理解成”拼一段 prompt 然后发请求”，而 codex-rs 的实现告诉我们，成熟的 agent 框架远不止如此。对于 Codex-RS 来说，一次 turn 在进入 loop 前至少要解决四类问题：

  • 本轮配置是什么，是否已经冻结。
  • 当前历史是否还能装下更多内容。
  • 模型应当看到哪些上下文、能力和用户输入。
  • 后续 loop 运行所需的通信与追踪对象是否已经就绪。

只有这些问题都处理完，模型采样才有意义。否则 loop 即使启动，也是在一个不稳定、信息不全、边界不清的环境里运行。

因此，如果要用一句话概括本文的结论，那就是：

run_turn() 真正的难点，不在”如何把模型跑起来”，而在”如何让模型在一个被准备好的世界里跑起来”。

而这个”被准备好的世界”，正是本文分析的进入 loop 前准备阶段。
