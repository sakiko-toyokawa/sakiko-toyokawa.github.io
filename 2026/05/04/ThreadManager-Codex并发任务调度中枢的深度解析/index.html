ThreadManager：Codex 并发任务调度中枢的深度解析

ThreadManager：Codex 并发任务调度中枢的深度解析

引言：当多个 AI 实例同时运行时

假设你的桌面端 Codex 同时开了三个对话窗口——一个在排查支付 API 的偶发超时，一个在重构用户模块的代码，一个在分析数据库迁移脚本。三个 AI 实例各自有自己的 Session 上下文、各自的 Tokio 后台任务、各自的 LLM 连接。

问题来了：当你在”支付 API”窗口里输入”检查 Redis 缓存超时配置”时，系统怎么知道这行指令应该路由给哪个 AI？怎么保证路由过程中不会因为目标线程已崩溃而丢指令？怎么在应用重启后恢复昨天的排查进度？

在上一篇文章中，我们拆解了 Session 内部的双轨制调度（steer_input / spawn_task）。但那个机制只管单个 Session 内部的热注入决策。当系统同时存在多个 Session 时，需要更上层的东西来管理”谁是谁”——这就是 ThreadManager。

ThreadManager 不是”调度器”（scheduler），它是物理层管理者。它维护 ThreadId → CodexThread 的映射表，负责 Thread 的创建、恢复、查找、路由、持久化和关闭。

一、核心结构：ThreadManager 的两个关键字段

1.1 ThreadManager 本体

ThreadManager 本身极简——它只是一个 Arc<ThreadManagerState> 的持有者。真正的逻辑和状态都在 ThreadManagerState 中：

从这两段代码可以读出 ThreadManager 的三大职责：

① 线程注册表：Arc<RwLock<HashMap<ThreadId, Arc<CodexThread>>>> — 一个支持并发读、写时互斥的 HashMap。它回答了”用 thread_id 找 AI 实例”这个最基础的问题。

② 事件广播：broadcast::Sender<ThreadId> — 每当有新线程创建，ThreadManager 会通过这个通道通知所有订阅方（UI 列表更新、日志审计、监控面板）。

③ 共享服务注入：models_manager、skills_manager、plugins_manager 等是全局共享的——不是每个 Thread 各自加载一套插件，而是所有 Thread 共享同一个插件管理器。这避免了重复加载数百个 TOCT 定义文件。

1.2 CodexThread：ThreadManager 管理的基本单元

每个 CodexThread 包着：

  • codex: Codex：即 Session，承载全部 AI 状态（对话历史、工具注册、沙箱策略）
  • session_source: SessionSource：标记这个 Thread 的来源——用户手动创建 / 子 Agent 派生 / CLI 启动
  • session_configured：Session 配置完成后的首条事件（前端靠它判断”可以开始交互了”）
  • rollout_path：对话录制文件的路径（用于崩溃恢复）

CodexThread 暴露的控制接口（codex_thread.rs L132–224）包括：

  • submit(op) → 转发到 self.codex.submit(op)
  • steer_input(...) → 转发到 self.codex.steer_input(...)
  • shutdown_and_wait() → 转发到 self.codex.shutdown_and_wait()

它是一个纯粹的外观层，不包含任何调度逻辑。

1.3 物理层与逻辑层的分层

ThreadManager 不关心 Session 内部在干什么——它只关心”这个 Thread 是否存在、是否活着、如何找到它”。

Codex::spawn 是 Codex 架构中最”I/O 密集”的函数——加载历史、解析插件 TOML、初始化 SQLite 连接、启动 submission_loop 后台 Task。ThreadManager 只需传参并等待它返回 CodexSpawnOk { codex, thread_id }。

2.3 finalize：注册入表

三个关键设计细节：

① 等待 SessionConfigured 才注册：确保 Thread 在进入注册表时已经完成了所有初始化（技能加载、环境检查）。外部通过 get_thread(id) 拿到的 Thread 一定是”就绪”的。

② Vacant 检查：threads.entry(thread_id) 使用 Entry API，保证”读-检查-插入”三步操作的原子性。如果 thread_id 已被占用（比如两个请求同时尝试创建同一个恢复线程），会走到 shutdown_and_wait 清理冗余实例。

③ 广播通知：finalize 完成后调用 notify_thread_created(thread_id)，通过 broadcast::Sender 推送新 Thread 的 ID。

三、Resume：恢复已有的对话

用户关闭应用后重新打开，如何恢复昨天的对话进度？resume_thread_with_history 的处理比”重新 spawn”多了一个关键步骤——去重检查。

这里的幂等性设计非常工程化：如果同一个 thread_id 的恢复请求来了两次，第二次直接返回已有的 Arc<CodexThread>——不会创建两个相同 ID 的线程。但如果旧的 Thread 已经崩溃退出（is_running() == false），就清理残留并重建。

恢复的数据来源是 ThreadStore，它从 SQLite 中读取历史快照。这部分在第六章展开。

六、持久化：ThreadStore 与内存-磁盘桥接

ThreadManager 不只是管理”内存中的线程”。它还持有一个 ThreadStore，负责将线程状态持久化到磁盘：

6.1 ThreadStore 的三种后端

  • Local：本地 SQLite（默认生产路径）
  • Remote：远程服务端点（分布式部署场景）
  • InMemory：纯内存（测试用，重启即丢失）

6.2 持久化的范围

ThreadManager 不直接持久化 Session 内部的每个 LLM token——那由 RolloutRecorder 负责。ThreadManager 持久化的是线程元数据——Thread 列表、Goal 状态、对话摘要——在应用重启时用于重建内存注册表。

6.3 冷启动恢复流程

应用启动时，恢复流程大致是：

  1. 扫描 ThreadStore 中所有未完成的 Thread
  2. 对每个 Thread，加载其 RolloutRecorder 录制文件
  3. 通过 resume_thread_with_history 逐个恢复
  4. spawn_thread_with_source 处理去重逻辑（已在运行的跳过 / 已崩溃的清理重建）

这保证了一个关键特性：应用重启后，昨天的对话进度不会丢失。

七、并行优雅关机

ThreadManager 的关机不是 for thread in threads { thread.shutdown().await } 的串行循环。它使用 FuturesUnordered 实现并行关机：

这五个关键设计：

  1. 先 Clone 再关机：在持有读锁期间 clone 出所有 Arc<CodexThread>，放下锁后再关机——防止关机的 await 阻塞其他读操作。
  2. FuturesUnordered：N 个线程的 shutdown 并行执行。如果有 10 个 Thread，每个需要 2 秒落盘 SQLite，并行关机总共约 2 秒而非 20 秒。
  3. TimeOut 保护：tokio::time::timeout(timeout, ...) 防止个别卡死的 Thread 阻塞整个关机流程。
  4. 分类报告：返回 ThreadShutdownReport { completed, submit_failed, timed_out } 而不是”成功/失败”二元结果——调用方可以根据分类做出不同处理（timed_out → 重试，submit_failed → 告警）。
  5. 清理注册表：已完成的 Thread 从 HashMap 中移除，超时/失败的保留——允许后续重试。

八、隔离性：一个 Thread 崩溃不会拖垮全局

ThreadManager 的注册表使用 Arc<CodexThread>——即 Thread 的生命周期由引用计数管理，不绑定注册表。这意味着：

  • HashMap 中的条目只是引用：如果某个 Thread 内部的 submission_loop panic 了，JoinHandle 完成，但 Arc<CodexThread> 仍然有效。外部的”线程死亡”检测通过 is_running() 实现：

  • 隔离的 Tokio Task 树：每个 CodexThread 通过 Codex::spawn 创建了独立的 submission_loop tokio task。这个 task 及其子 task 通过 CancellationToken 形成一棵独立的取消树——A 线程的 cancel 不会影响 B 线程。

  • RwLock 保证查找不受创建阻塞：读线程列表用读锁（多并发），创建新线程用写锁（互斥）。正常的”查找-路由”操作不会被其他线程的 spawn 阻塞（除非恰好冲突到同一个锁周期）。

  • AgentControl 弱引用：ThreadManagerState 通过 Arc::downgrade 传递自身的弱引用给 AgentControl，避免循环引用导致的内存泄漏：

九、跨语言对比

如果不用 Rust + Actor，ThreadManager 的职责在其它语言中如何实现？

语言/框架 等价实现 核心缺陷
Go map[string]*goroutine + sync.RWMutex GC STW 可能在频繁创建/销毁 Thread 时产生毛刺；*goroutine 无法真正”管理”goroutine——无法等待其结束，无法传递 CancellationToken
Python dict[str, asyncio.Task] + asyncio.Lock GIL 让”并行关机”（FuturesUnordered 的真正并行）变成伪命题
Node.js Map<string, Worker> + worker_threads Worker 进程开销大（每线程独立 V8 实例），上百个对话窗口不可行
Erlang 原生进程注册表 (register()) 进程崩溃后 OTP Supervisor 自动重启——这一点比 Codex 更优雅。但缺乏 Tokio 的细粒度 CancellationToken 树取消
Rust (Codex) Arc<RwLock<HashMap<ThreadId, CodexThread>>> + FuturesUnordered 编译期线程安全 + 零成本抽象 + 跨平台文件系统沙箱

Rust 的组合优势体现在三个地方：Arc<RwLock<HashMap>> 的类型签名本身就是编译期强制执行的并发协议；FuturesUnordered 的并行关机是真正的 OS 级并发（不锁在同一个事件循环里）；CancellationToken 的树取消让每个 Thread 内部的子 Task 能被精确清理。

十、总结

回顾 ThreadManager 的设计，可以提炼三条核心原则：

1. 分层管理 —— ThreadManager 管”谁活着”，CodexThread 管”怎么交互”，Session 管”在想什么”。三层职责明确，互不越界。

2. 幂等恢复 —— Resume 时的去重检查保证同一 thread_id 不会产生两个活跃实例。幂等性不是”附加特性”，而是并发系统中的必要条件。

3. 并行而非串行 —— 无论是注册表的并发读（RwLock）还是关机的并发执行（FuturesUnordered），ThreadManager 从不假设”一次只做一件事”。

如果说 Session 是 Codex 的”大脑”，ThreadManager 就是 Codex 的”躯体”——它让多个大脑能同时工作而不打架，能在休克（崩溃）后苏醒（恢复），能在需要休眠（关机）时整齐地逐一完成末了之事。
