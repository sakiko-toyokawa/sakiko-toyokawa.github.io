Codex 双轨制任务驱动架构：从 steer_input 到 spawn_task 的深度解析

引言：当用户在终端里按下回车

假设你正在使用 Codex 排查一个棘手的问题——“我们支付服务的 API 偶发 504 超时”。Codex 已经在你的代码仓库里搜索了日志模式、阅读了超时中间件的源码、分析了 Nginx 配置。它的推理过程正在终端上逐字渲染。

这时你突然想到：会不会是数据库连接池耗尽导致的？ 你在 TUI 中输入这行字，敲下回车。

这一刻，Codex 内部发生了一件极度精妙的事。它没有中断当前的推理去”接收新指令”——那个正在运行的 AI 思考线程不需要重启，不需要取消重来。你的新输入被静默注入到了正在执行的任务中，AI 会在当前轮次的下一个循环自动读到它。

这就是 Codex 的双轨制任务驱动架构。

本文基于 codex-rs/core/src/ 的真实源码，逐行拆解这一机制的底层实现。全文将以”排查 API 偶发超时”为贯穿示例。

在进入双轨调度逻辑之前，先厘清三个核心组件。

这是全文的核心章节。当用户的 Op 到达 submission_loop，match 到 Op::UserInput 分支后，会调用 user_input_or_turn。它自身只是一个薄壳：

真正的逻辑在 user_input_or_turn_inner 中。它的职责分三段：

2.1 第一段：解析 Op（L131–247）

将 Op::UserTurn / Op::UserInputWithTurnContext / Op::UserInput 三种变体统一解析为 (items, updates, metadata) 三元组。这一步处理了大量配置合并逻辑（协作模式、权限策略、沙箱级别、模型选择等），我们略过细节。

2.2 第二段：创建 TurnContext（L249–254）

new_turn_with_sub_id 创建一个新的 TurnContext——包含 sub_id、模型信息、遥测状态、协作模式等元数据。无论最终走热注入还是冷启动，这个 Context 都会被用到。

2.3 第三段：双轨分叉（L255–291）——全文最关键代码

这个 match 就是双轨制的全部秘密。三条分支对应三种场景：

分支 条件 含义 用户感知
Ok(_) AI 正在运行 Regular Task 热注入：输入塞入信箱 “AI 收到我的纠正，继续工作”
Err(NoActiveTurn(items)) AI 当前空闲 冷启动：发起新 Task “AI 开始排查我的问题”
Err(other) 状态不匹配/类型冲突 透传错误事件 “操作被拒绝，原因是…”

用我们的示例来解释：

  • 场景 A（热注入）：Codex 正在搜索日志模式时，你输入”查一下数据库连接池配置”→ steer_input 成功 → 输入注入当前 Turn 的信箱 → 当前 RegularTask 下一圈 run_turn 会读到它。

  • 场景 B（冷启动）：Codex 刚启动，你输入”排查这个 API 为什么偶发超时”→ steer_input 返回 NoActiveTurn → 调用 spawn_task(RegularTask::new()) → 全新 Tokio Task 被创建，AI 开始工作。

  • 场景 C（错误）：Codex 正在执行代码审查（Review Task），你试图干预 → 返回 ActiveTurnNotSteerable → 前端收到错误提示。

三、热注入：steer_input 的五道关卡

steer_input 是双轨调度中的”评委”。它不启动任何任务，只做判断和投递。它设在 Session 上，源码在 session/mod.rs L2963–3019。我们逐关拆解。

关卡 1：空输入检测

不废话，空输入直接拒绝。SteerInputError 是一个枚举（见第七章），每个错误变体都在上层有对应的处理路径。

关卡 2：活跃任务存在性

获取 active_turn 的互斥锁。如果为 None，返回 NoActiveTurn——携带着原始 input，不丢失数据。这是”错误即逻辑”模式的核心：上层 match 到这个错误就直接走冷启动路径。

关卡 3：Turn ID 校验

调用方可以指定 expected_turn_id 来确保”我干预的就是我以为的那个 Turn”。在 user_input_or_turn_inner 中传的是 None——不校验。但在多线程并发操作同一 Session 的场景中，这个校验是并发安全的关键守护。

关卡 4：TaskKind 守卫（关键！）

三种 TaskKind 只有 Regular 允许被用户干预。Review（代码审查）和 Compact（上下文压缩）是”敏感手术”——干预可能破坏内部逻辑的原子性。这一点在第六章会展开。

关卡 5：投递到信箱

这是热注入的”落点”。两步操作：

  1. push_pending_input(input.into())——用户输入存入 TurnState.pending_input
  2. accept_mailbox_delivery_for_current_turn()——强制将邮箱状态机切回 CurrentTurn，确保子 Agent 邮件也能被当前 Turn 消费

两步之间是一个原子操作（在同一个 turn_state 锁内），避免了”推入了输入但邮箱状态不匹配”的竞态。

这意味着一个 RegularTask 的生命周期可能跨越多轮用户交互：

  1. 用户输入”排查 API 超时” → 冷启动 RegularTask → run_turn(输入) 执行 → LLM 搜索日志、读取源码
  2. 用户中途输入”查数据库连接池” → 热注入 → has_pending_input() == true → 不退出

这种设计避免了”每次用户纠正都创建新 Task”的开销，同时保持了逻辑的清晰：一个 Task = 一个连续的用户意图演化过程。

六、任务类型守卫：为什么 Review/Compact 不可干预

Codex 定义了三种任务类型：

steer_input 只对 Regular 放行。这个设计不是防御性的——它有明确的语义含义：

Review 任务：AI 正在进行一次完整的代码审查。审查过程会建立对代码库的连贯理解，中途插入新指令会破坏审查上下文的完整性。想象你在 code review 到一半时有人拍了拍你说”顺便把那个也修了”——结果可能是两个都没做好。

Compact 任务：AI 正在压缩对话上下文（将早期消息转为摘要以防止 token 爆炸）。这是一个内部维护操作，压缩过程需要精确知道”哪些历史需要保留”——此时如果有新输入进来，可能导致压缩的边界条件出错，丢失关键历史。

这个守卫的实现方式值得注意——用的是 Rust 的模式匹配穷尽检查（match on enum），而不是运行时 if/else：

如果将来有人新增了 TaskKind::BackgroundWarmup 变体，编译器会在编译期报错，迫使开发者显式决定”这个新类型能不能被用户干预”。而不是悄悄漏过去。

  • 不抛异常——用 Result 表达控制流
  • 错误变体携带上下文——调用方不需要额外存储输入
  • 一次锁操作完成判断+投递——消除了 check-then-act 竞态

八、跨语言对比

同一套双轨调度逻辑，在不同语言生态中的实现难度差异显著：

语言/框架 等价实现 核心缺陷
Python + AsyncIO asyncio.Queue + asyncio.create_task GIL 限制真正的并行工具调用；active_turn 状态无编译期保护，多协程可能同时写入
Go goroutine + select + channel 无 affine type——Mutex<Option<ActiveTurn>> 在 Go 中靠 convention 保证不复制，无法在编译期发现误用
Node.js + TypeScript EventEmitter + 全局状态标志 单线程——但大文件 diff 或 Shell 命令的 stdout 解析会阻塞事件循环
Erlang / Elixir GenServer + 信箱 Actor 模型原生支持消息注入，但缺乏 Tokio 的 CancellationToken 树取消能力
Rust (Codex) Actor + Mutex<Option<ActiveTurn>> + pending_input 编译期所有权 + 零开销抽象，但学习曲线陡峭

Rust 的杀手级组合在于：Mutex<Option<ActiveTurn>> 的类型签名本身就是一份编译期强制执行的协议。你不会忘记加锁，不会忘记检查 None，不会在线程间传递没有 Send 的数据。这些在其他语言中需要靠代码规范和 code review 维持的纪律，在 Rust 中是编译器说了算。

关键设计点

  1. Mutex<Option<TurnState>>：Option 直接编码了”是否有活跃任务”的状态——不需要额外的 bool 标志。
  2. Error-as-Logic：steer_input 返回 Result，Err(NoActiveTurn(items)) 携带原始输入，无损转为 spawn_task 参数。
  3. std::mem::take：在消费信箱时用 swap 替代 clone，零拷贝且保持锁持有时间最短。
  4. Task 生命周期：一个 Turn 可以执行多个 cycle（只要信箱有货），模拟了真实 RegularTask 的 loop 行为。

十、总结：三条核心原则

回顾 Codex 双轨调度架构的精髓：

1. 错误即逻辑 —— steer_input 失败不是异常，是控制流信息。NoActiveTurn 携带原始输入，无缝衔接冷启动。这消除了”先检查再操作”的竞态窗口。

2. 类型守卫 —— TaskKind 的 match 穷尽检查，让编译器保证”不会漏掉对新任务类型的干预权限决策”。Review/Compact 的不允许干预不是注释里写的约定，是编译期强制执行的规则。

3. 邮箱解耦 —— TurnState.pending_input 让用户输入的生产和消费完全异步。生产者只需 push，消费者在 loop 中自然检查。加上 MailboxDeliveryPhase 状态机，子 Agent 间的通信也被纳入同一套调度体系。

从用户敲下回车到 AI 做出响应，Codex 没有走”收到请求→创建线程→执行→返回”的简单路径。它问了一个更聪明的问题：这个意图是应该注入正在燃烧的火焰，还是点燃一根新的火柴？
